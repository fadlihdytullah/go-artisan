<Section num="01" id="intro" title="Kenapa Networking Penting?" sub="Jembatan antara resource, user, kantor, dan internet">

<p class="lead">Aplikasi cloud tidak hidup sendirian, ia harus bisa dijangkau, diamankan, dan diarahkan dengan benar.</p>

Analogi paling sederhana: sebuah toko online butuh alamat toko, jalan masuk, satpam, gudang belakang, kurir cepat, dan jalur khusus ke kantor pusat. Di AWS, peran itu tersebar ke beberapa layanan networking. **VPC** adalah area jaringan privat, **subnet** adalah blok kawasan, **route table** adalah papan arah, **security group** dan **NACL** adalah lapisan keamanan, **Route 53** adalah DNS, **CloudFront** adalah CDN, sedangkan **VPN** dan **Direct Connect** adalah jalur dari kantor atau data center ke AWS.

Untuk CLF-C02, kamu tidak perlu menghafal desain routing sedalam Solutions Architect. Fokus utamanya adalah: tahu fungsi layanan, bisa memilih layanan yang tepat dari skenario, dan tidak tertukar antara DNS, CDN, private network, hybrid connection, dan network security. Inilah kenapa modul ini berkali-kali memakai pola "X vs Y". Soal networking di CLF-C02 jarang menguji syntax, tetapi sangat sering menguji apakah kamu bisa membedakan dua layanan yang sekilas mirip, misalnya security group vs NACL, CloudFront vs Global Accelerator, atau gateway endpoint vs interface endpoint.

Kenapa networking jadi tulang punggung Domain 3? Karena hampir semua layanan AWS lain (EC2, RDS, Lambda di VPC, ECS) berdiri di atas jaringan. Salah pasang security group bisa membuat aplikasi tidak bisa diakses; salah taruh database di public subnet bisa membuka celah keamanan; salah pilih CloudFront padahal yang dibutuhkan static IP global bisa membuat solusi tidak jalan. Jadi memahami networking bukan sekadar menambah satu topik, tetapi memahami fondasi yang menopang topik lain.

<Box variant="analogy" icon="🏙️" label="Analogi kota"><p>VPC seperti komplek perumahan berpagar, subnet seperti blok rumah, route table seperti rambu jalan, security group seperti kunci pintu tiap rumah, NACL seperti gerbang blok, Route 53 seperti buku alamat, dan CloudFront seperti cabang toko yang lebih dekat ke pembeli.</p></Box>

<p>Menurut <a href="https://docs.aws.amazon.com/aws-certification/latest/cloud-practitioner-02/cloud-practitioner-02.html">Exam Guide CLF-C02</a>, layanan networking dan content delivery masuk Domain 3 (Cloud Technology and Services, 34%), domain dengan bobot terbesar. Tetapi modul ini juga menyentuh Domain 2 (security group, NACL, private subnet, endpoint policy) dan Domain 4 (biaya NAT Gateway, data transfer, dan model harga CloudFront), sehingga satu topik bisa menjawab beberapa domain sekaligus.</p>

<Box variant="exam" icon="🎯" label="Fokus ujian"><p>Soal CLF-C02 biasanya menanyakan layanan mana yang cocok, bukan cara menulis route table kompleks atau menghitung CIDR secara detail. Latih dirimu mengubah kalimat soal menjadi satu kata kunci, lalu petakan kata kunci itu ke satu layanan.</p></Box>

</Section>

<Section num="02" id="peta-besar" title="Peta Besar Layanan Networking" sub="Kelompokkan dulu supaya tidak terasa penuh">

<p class="lead">Networking AWS lebih mudah dipahami jika dibagi menjadi empat kelompok: fondasi jaringan, keamanan, edge, dan koneksi hybrid.</p>

AWS memasukkan layanan networking dan content delivery seperti **Amazon VPC**, **Amazon Route 53**, **Amazon CloudFront**, **AWS Direct Connect**, **AWS VPN**, **AWS Site-to-Site VPN**, **AWS Client VPN**, **AWS PrivateLink**, **AWS Transit Gateway**, **AWS Global Accelerator**, dan **Amazon API Gateway** dalam cakupan CLF-C02. Artinya, pemula perlu mengenali tujuan layanan ini, bukan menguasai semua opsi lanjutan. Daftar resmi lengkap ada di [CLF-C02 In-Scope Services](https://docs.aws.amazon.com/aws-certification/latest/cloud-practitioner-02/clf-02-in-scope-services.html).

<Box variant="tip" icon="🗂️" label="AWS VPN itu payung"><p>Di exam guide, AWS VPN, AWS Site-to-Site VPN, dan AWS Client VPN ditulis sebagai tiga entri terpisah. Ingat ini: <strong>AWS VPN</strong> adalah nama payung, sedangkan <strong>Site-to-Site VPN</strong> (kantor atau data center ke AWS) dan <strong>Client VPN</strong> (user remote ke VPC) adalah dua produk di bawahnya. Jadi jangan bingung jika ketiganya muncul sebagai pilihan terpisah.</p></Box>

<Section num="03" id="vpc-dasar" title="VPC, Subnet, dan Route Table" sub="Fondasi private network di AWS">

<p class="lead">VPC adalah jaringan virtual yang kamu definisikan sendiri di dalam AWS account.</p>

**Amazon VPC** memungkinkan kamu menjalankan resource AWS di jaringan yang terisolasi secara logis. Kamu memilih rentang alamat IP, membagi jaringan menjadi subnet, mengatur rute, dan menempelkan kontrol keamanan. Untuk pemula, VPC bisa dianggap sebagai tanah kavling privat tempat kamu membangun rumah aplikasi.

**Subnet** adalah potongan rentang IP di dalam VPC. Satu subnet selalu hidup di satu Availability Zone saja, ia tidak bisa membentang ke dua AZ. Ini penting karena arsitektur yang tahan gangguan menyebar resource ke lebih dari satu Availability Zone, jadi kamu biasanya membuat satu subnet per AZ. **Route table** menentukan ke mana traffic dari subnet diarahkan. Inilah inti yang sering jadi jebakan: yang membuat subnet "public" atau "private" bukan namanya, tetapi isi route table-nya. Jika route table punya rute ke internet gateway, subnet tersebut menjadi public subnet. Jika tidak punya rute langsung ke internet gateway, subnet tersebut adalah private subnet.

Kenapa pembagian ini ada? Bayangkan kamu tidak ingin semua resource bisa dijangkau langsung dari internet. Dengan memisahkan subnet dan mengatur route table, kamu menentukan mana yang boleh "menghadap jalan raya" (public) dan mana yang harus "di ruang belakang" (private). Konsep ini menjadi pondasi keamanan jaringan di AWS.

<h3>Komponen yang sering muncul di skenario</h3>

<ul>
  <li><strong>Internet Gateway</strong>, gerbang agar resource di public subnet bisa berkomunikasi dengan internet.</li>
  <li><strong>NAT Gateway</strong>, jalan keluar untuk resource di private subnet menuju internet tanpa menerima koneksi masuk langsung dari internet.</li>
  <li><strong>Route Table</strong>, pengarah traffic berdasarkan destination dan target.</li>
  <li><strong>VPC Flow Logs</strong>, catatan metadata traffic IP yang berguna untuk observability dan troubleshooting.</li>
</ul>

<Box variant="exam" icon="🧠" label="Jebakan ujian"><p>Subnet disebut public bukan karena namanya mengandung kata public, tetapi karena route table subnet itu mengarah ke internet gateway dan resource terkait punya konfigurasi publik yang sesuai.</p></Box>

</Section>

<Section num="04" id="public-private" title="Public Subnet, Private Subnet, dan NAT" sub="Memisahkan pintu depan dan ruang belakang aplikasi">

<p class="lead">Desain network yang umum: hanya komponen depan yang publik, komponen belakang tetap privat.</p>

Bayangkan restoran. Area kasir boleh terlihat pelanggan, tetapi dapur, gudang, dan brankas tidak boleh langsung diakses dari jalan. Di AWS, load balancer bisa berada di public subnet untuk menerima traffic user, sedangkan EC2 application server dan database berada di private subnet.

<Steps>
  <Step><b>User mengakses domain</b><p>User mengetik domain, DNS mengarah ke CloudFront, load balancer, atau endpoint aplikasi.</p></Step>
  <Step><b>Traffic masuk lewat komponen publik</b><p>Public subnet biasanya menampung Application Load Balancer atau bastion yang benar-benar diperlukan.</p></Step>
  <Step><b>Aplikasi berjalan di private subnet</b><p>EC2, container, atau database tidak perlu punya public IP jika hanya melayani traffic internal.</p></Step>
  <Step><b>Private resource keluar lewat NAT atau endpoint</b><p>Untuk update package atau akses layanan AWS, private subnet bisa memakai NAT gateway atau VPC endpoint sesuai kebutuhan.</p></Step>
</Steps>

<h3>Internet Gateway vs NAT Gateway, beda arah traffic</h3>

<p>Dua komponen ini paling sering tertukar karena sama-sama "menghubungkan ke internet". Bedanya ada di arah traffic. Internet Gateway adalah pintu dua arah untuk public subnet: traffic dari internet boleh masuk, traffic ke internet boleh keluar. NAT Gateway hanya satu arah: resource di private subnet boleh keluar ke internet (misalnya mengunduh update), tetapi internet tidak bisa memulai koneksi masuk ke resource itu.</p>

<Compare aLabel="Internet Gateway &middot; dua arah" bLabel="NAT Gateway &middot; outbound saja" aTone="blue" bTone="teal" vs="vs">
  <Fragment slot="a"><ul><li>Dipasang di VPC, menjadi gerbang public subnet ke internet.</li><li>Mengizinkan koneksi masuk (inbound) maupun keluar (outbound).</li><li>Cocok untuk ALB publik atau resource yang memang harus dijangkau dari internet.</li><li>Tidak menambah biaya per-jam tersendiri untuk gateway-nya.</li></ul></Fragment>
  <Fragment slot="b"><ul><li>Dipasang di public subnet, melayani resource di private subnet.</li><li>Hanya mengizinkan koneksi keluar (outbound), tidak menerima inbound dari internet.</li><li>Cocok agar EC2 private bisa update package tanpa membuka diri ke internet.</li><li>Berbiaya per-jam ditambah per-GB data yang diproses, jadi perlu diperhitungkan.</li></ul></Fragment>
</Compare>

<Box variant="analogy" icon="🚪" label="Analogi pintu"><p>Internet Gateway seperti pintu depan toko yang bisa dilewati dua arah, pelanggan masuk dan keluar. NAT Gateway seperti pintu belakang yang hanya bisa dibuka dari dalam: karyawan boleh keluar mengambil barang, tetapi orang luar tidak bisa masuk lewat situ.</p></Box>

<Box variant="tip" icon="💰" label="Catatan biaya NAT"><p>NAT Gateway berbiaya per-jam plus per-GB data yang diproses, jadi sering jadi pertimbangan cost. Salah satu trik hemat: untuk akses ke S3 atau DynamoDB, gateway endpoint (gratis) bisa menggantikan jalur lewat NAT Gateway. Selalu cek <a href="https://aws.amazon.com/vpc/pricing/">AWS Pricing</a> untuk angka terbaru, jangan menghafal nominalnya untuk ujian.</p></Box>

<Box variant="warn" icon="⚠️" label="Kesalahan pemula"><p>Menaruh database di public subnet karena ingin mudah dites adalah pola berisiko. Untuk skenario ujian, database biasanya berada di private subnet.</p></Box>

</Section>

<Section num="05" id="network-security" title="Security Group vs Network ACL" sub="Kunci pintu resource vs gerbang subnet">

<p class="lead">Security group dan NACL sama-sama firewall, tetapi bekerja di level yang berbeda.</p>

**Security group** mengontrol traffic inbound dan outbound untuk resource seperti EC2 atau load balancer. Ia bersifat **stateful**, artinya jika request diizinkan keluar, response yang kembali otomatis diizinkan. **Network ACL** bekerja di level subnet. Ia bersifat **stateless**, artinya aturan inbound dan outbound harus dipikirkan dua arah.

Kenapa "stateful" begitu penting? Karena membuat hidup lebih mudah. Pada security group, kamu cukup mengizinkan request keluar, maka jawabannya otomatis boleh kembali tanpa kamu menulis rule balikan. Pada NACL yang stateless, AWS tidak mengingat koneksi, jadi kamu harus eksplisit mengizinkan return traffic, biasanya pada rentang port ephemeral. Inilah sumber bug klasik pada NACL: lupa membuka return traffic, lalu koneksi terasa "setengah jalan".

<Box variant="tip" icon="🔍" label="Observasi di Console"><p>Untuk membiasakan diri, di AWS Console buka VPC &rarr; Subnets, pilih sebuah subnet, lalu cek tab Route table. Di EC2 &rarr; Security Groups kamu bisa melihat hanya ada tab Inbound dan Outbound rules tanpa kolom deny, itu menegaskan sifat allow-only. Langkah baca ini aman dan tidak menambah biaya.</p></Box>

<Box variant="exam" icon="🎯" label="Pola soal favorit"><p>Jika soal menanyakan firewall stateful di level instance, jawab security group. Jika menanyakan firewall stateless di level subnet dengan allow dan deny, jawab network ACL.</p></Box>

<Section num="06" id="endpoint-privatelink" title="VPC Endpoint dan AWS PrivateLink" sub="Akses privat ke layanan tanpa lewat internet publik">

<p class="lead">VPC endpoint adalah pintu privat dari VPC ke layanan, PrivateLink adalah teknologi koneksi privat untuk banyak skenario endpoint.</p>

Tanpa endpoint, resource di private subnet sering perlu NAT gateway untuk mengakses layanan AWS seperti S3 melalui public endpoint. Dengan **VPC endpoint**, traffic bisa diarahkan secara privat dari VPC ke layanan yang didukung. Ini mengurangi kebutuhan melewati internet publik dan membantu kontrol keamanan.

Ada dua istilah yang sering tertukar. **VPC endpoint** adalah resource yang kamu buat di VPC. **AWS PrivateLink** adalah teknologi yang memungkinkan resource di VPC terhubung ke layanan AWS memakai private IP, seolah-olah layanan tersebut di-host di dalam VPC kamu, tanpa internet gateway atau NAT device. PrivateLink dipakai oleh interface endpoint untuk menjangkau layanan AWS, layanan milik account lain, atau layanan partner secara privat.

<CardGrid cols={2}>
  <Card><h4>Gateway endpoint</h4><p>Dipakai untuk Amazon S3 dan DynamoDB. Route table akan mendapatkan rute ke endpoint, gratis tanpa biaya tambahan, dan tidak memakai AWS PrivateLink.</p></Card>
  <Card><h4>Interface endpoint</h4><p>Memakai elastic network interface dengan private IP di subnet, memakai AWS PrivateLink, dan berbayar (per-jam per-AZ ditambah per-GB data processing).</p></Card>
</CardGrid>

<Figure><VPCNetworkingFig05 /><Fragment slot="caption"><b>Gambar 5.</b> Gateway endpoint (gratis, lewat route table, hanya S3 dan DynamoDB) dibandingkan interface endpoint (berbayar, lewat ENI dan private IP, memakai PrivateLink, banyak layanan).</Fragment></Figure>

<Compare aLabel="Gateway endpoint" bLabel="Interface endpoint (PrivateLink)" aTone="teal" bTone="blue" vs="vs">
  <Fragment slot="a"><ul><li>Hanya untuk Amazon S3 dan DynamoDB.</li><li>Gratis, tidak ada biaya tambahan.</li><li>Bekerja lewat entri di route table (prefix list).</li><li>Bukan teknologi PrivateLink.</li></ul></Fragment>
  <Fragment slot="b"><ul><li>Mendukung banyak layanan AWS dan layanan partner.</li><li>Berbayar per-jam per-AZ ditambah per-GB data processing.</li><li>Bekerja lewat elastic network interface dengan private IP di subnet.</li><li>Memakai teknologi AWS PrivateLink.</li></ul></Fragment>
</Compare>

<h3>Kapan dipakai?</h3>

<ul>
  <li>EC2 di private subnet perlu mengambil objek dari S3 tanpa NAT gateway.</li>
  <li>Aplikasi internal perlu memanggil layanan AWS memakai private IP.</li>
  <li>Perusahaan ingin mengekspos layanan antar VPC atau antar account tanpa membuka akses publik.</li>
  <li>Tim keamanan ingin membatasi akses memakai endpoint policy, bucket policy, atau kondisi seperti VPC endpoint tertentu.</li>
</ul>

<Box variant="tip" icon="💡" label="Cara memilih"><p>Untuk S3 atau DynamoDB dari VPC yang sama, gateway endpoint sering menjadi jawaban ujian. Untuk banyak layanan AWS lain atau layanan partner, pikirkan interface endpoint dan PrivateLink.</p></Box>

<Box variant="warn" icon="⚠️" label="Jangan disamakan"><p>PrivateLink bukan VPN dan bukan Direct Connect. PrivateLink adalah koneksi privat ke layanan, sedangkan VPN dan Direct Connect menghubungkan network seperti kantor atau data center ke AWS.</p></Box>

</Section>

<Section num="07" id="dns-route53" title="DNS dengan Amazon Route 53" sub="Buku alamat untuk aplikasi cloud">

<p class="lead">Route 53 membantu user menemukan endpoint aplikasi melalui nama domain yang mudah diingat.</p>

**DNS** menerjemahkan nama seperti `www.example.com` menjadi alamat tujuan yang bisa dipakai jaringan. **[Amazon Route 53](https://docs.aws.amazon.com/Route53/latest/DeveloperGuide/Welcome.html)** adalah layanan DNS yang highly available dan scalable. Ia dapat dipakai untuk registrasi domain, DNS routing, dan health checking. Nama "53" mengacu pada port 53, port standar protokol DNS, sekaligus cara mudah mengingat bahwa ini layanan DNS. Di skenario ujian, Route 53 sering muncul saat aplikasi perlu diarahkan ke load balancer, CloudFront distribution, atau endpoint sehat berdasarkan kebijakan routing.

<CardGrid cols={3}>
  <Card><h4>Domain registration</h4><p>Mendaftarkan nama domain seperti `example.com` melalui Route 53 jika kebutuhan bisnis mengarah ke sana.</p></Card>
  <Card><h4>DNS routing</h4><p>Mengarahkan nama domain ke resource AWS atau non-AWS menggunakan record DNS.</p></Card>
  <Card><h4>Health checks</h4><p>Memantau endpoint dan membantu mengalihkan traffic dari resource yang tidak sehat.</p></Card>
</CardGrid>

<h3>Routing policy yang perlu dikenali</h3>

<p>Route 53 punya delapan routing policy. Lima yang pertama paling sering muncul, tiga sisanya cukup kamu kenali namanya agar tidak salah jika muncul sebagai distractor. Daftar resmi ada di <a href="https://docs.aws.amazon.com/Route53/latest/DeveloperGuide/routing-policy.html">dokumentasi routing policy Route 53</a>.</p>

<div class="tbl-wrap"><table>

</Section>

<Section num="08" id="cdn-cloudfront" title="CDN dengan Amazon CloudFront" sub="Mendekatkan konten ke user">

<p class="lead">CloudFront mempercepat distribusi konten dengan menyimpan salinan di edge location yang dekat dengan user.</p>

Jika server aplikasi ada di satu Region, user yang jauh bisa merasakan latency lebih tinggi. **[Amazon CloudFront](https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/Introduction.html)** adalah CDN yang mempercepat pengiriman konten statis dan dinamis seperti HTML, CSS, JavaScript, gambar, video, API, atau aplikasi. CloudFront memakai jaringan edge global. Saat user meminta konten, request diarahkan ke edge location dengan latency rendah. Jika konten ada di cache, user mendapat response lebih cepat dan origin menerima beban lebih ringan.

Di balik edge location ada lapisan kedua bernama **Regional Edge Cache**, cache menengah berukuran lebih besar yang berada di antara edge location dan origin. Jika sebuah objek tidak ada di edge location tetapi masih tersimpan di Regional Edge Cache, CloudFront mengambilnya dari sana sebelum repot menghubungi origin. Detail ini opsional untuk CLF-C02, tetapi membantu kamu memahami kenapa cache hit rate CloudFront bisa tinggi.

<Steps>
  <Step><b>User meminta konten</b><p>Browser meminta file, API, atau halaman lewat domain yang mengarah ke CloudFront distribution.</p></Step>
  <Step><b>Edge location mengecek cache</b><p>Jika objek masih valid di cache, CloudFront mengirim response langsung dari edge.</p></Step>
  <Step><b>CloudFront mengambil dari origin</b><p>Jika cache miss, CloudFront mengambil konten dari origin seperti S3, ALB, EC2, atau HTTP server.</p></Step>
  <Step><b>Konten berikutnya lebih cepat</b><p>Salinan yang cacheable disimpan sesuai aturan cache agar request berikutnya lebih efisien.</p></Step>
</Steps>

<CardGrid cols={2}>
  <Card><h4>Origin</h4><p>Sumber konten, misalnya S3 bucket, Application Load Balancer, EC2, atau server HTTP lain.</p></Card>
  <Card><h4>Distribution</h4><p>Konfigurasi CloudFront yang mengatur domain, origin, cache behavior, TLS, logging, dan aturan delivery.</p></Card>
  <Card><h4>Edge location</h4><p>Lokasi jaringan global tempat CloudFront menerima request user dan menyajikan konten dengan latency rendah.</p></Card>
  <Card><h4>Cache behavior</h4><p>Aturan yang menentukan path, metode HTTP, header, cookie, query string, TTL, dan origin yang dipakai.</p></Card>
</CardGrid>

<Box variant="tip" icon="💡" label="Keamanan edge"><p>CloudFront sering dipasangkan dengan AWS WAF, AWS Shield (Standard otomatis aktif untuk proteksi DDoS), TLS certificate dari ACM, dan origin access control untuk membatasi akses langsung ke origin. Lihat juga modul Security untuk peran WAF dan Shield secara lebih dalam.</p></Box>

<h3>OAC vs OAI, jangan bingung melihat materi lama</h3>

<p>Untuk membatasi akses ke origin Amazon S3 agar hanya CloudFront yang boleh membaca bucket, ada dua mekanisme. <strong>Origin Access Control (OAC)</strong> adalah pendekatan terkini yang direkomendasikan AWS: mendukung semua Region, enkripsi SSE-KMS, dan request SigV4. <strong>Origin Access Identity (OAI)</strong> adalah metode legacy yang masih berfungsi pada distribusi lama, tetapi tidak lagi direkomendasikan. Jika kamu menemui istilah OAI di materi lama atau soal, ingat bahwa OAC adalah penggantinya. Detail ada di <a href="https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/private-content-restricting-access-to-s3.html">dokumentasi membatasi akses ke S3</a>.</p>

<Box variant="tip" icon="🎁" label="Free tier dan model harga"><p>CloudFront punya free tier yang sekarang "always free" (tidak terbatas 12 bulan pertama): 1 TB data transfer out dan 10 juta HTTP/HTTPS request per bulan. Selain pay-as-you-go, AWS juga memperkenalkan <a href="https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/flat-rate-pricing-plan.html">flat-rate pricing plans</a> (Free, Pro, Business, Premium) yang menggabungkan CDN, AWS WAF, Route 53, TLS, dan kredit lain tanpa overage charge. Untuk ujian, cukup tahu bahwa CloudFront punya free tier besar dan beberapa model harga, bukan menghafal angkanya.</p></Box>

<Box variant="exam" icon="🧠" label="Pola soal favorit"><p>Jika soal menekankan cache konten global, static website, video, file download, atau mengurangi latency pengiriman konten, CloudFront biasanya kandidat utama.</p></Box>

</Section>

<Section num="09" id="peering-transit" title="VPC Peering vs Transit Gateway" sub="Menghubungkan VPC, dari satu lawan satu hingga hub pusat">

<p class="lead">Saat aplikasi tumbuh, satu VPC sering tidak cukup. Pertanyaannya: bagaimana banyak VPC saling bicara dengan rapi?</p>

Bayangkan dua kantor yang ingin saling kirim dokumen. Cara termudah adalah membuat jalur langsung antara keduanya. Itulah **VPC Peering**: koneksi privat satu lawan satu antara dua VPC, sehingga resource di kedua VPC bisa berkomunikasi memakai private IP seolah berada di jaringan yang sama.

Masalah muncul saat kantornya bertambah banyak. VPC Peering bersifat **non-transitif**. Jika VPC A ter-peering dengan B, dan A ter-peering dengan C, maka B tetap tidak bisa menjangkau C lewat A. Kamu harus membuat peering langsung B ke C. Untuk empat VPC yang semuanya perlu saling terhubung, kamu butuh enam koneksi (rumus n(n-1)/2), dan angkanya meledak seiring jumlah VPC. Inilah yang disebut topologi full mesh, rumit untuk dikelola.

**[AWS Transit Gateway](https://docs.aws.amazon.com/vpc/latest/tgw/what-is-transit-gateway.html)** menyelesaikan kerumitan itu dengan menjadi hub pusat. Setiap VPC dan koneksi on-premises cukup attach satu kali ke hub, lalu Transit Gateway mengatur routing antar semuanya. Karena mendukung **transitive routing** (hub-and-spoke), VPC B bisa menjangkau VPC C melalui hub tanpa peering langsung. Satu hub menggantikan jaring koneksi yang ruwet.

<Figure><VPCNetworkingFig06 /><Fragment slot="caption"><b>Gambar 6.</b> VPC Peering membutuhkan full mesh (4 VPC = 6 koneksi, non-transitif), sedangkan Transit Gateway memakai hub-and-spoke (tiap VPC cukup satu attachment, transitive).</Fragment></Figure>

<Compare aLabel="VPC Peering &middot; satu lawan satu" bLabel="Transit Gateway &middot; hub-and-spoke" aTone="red" bTone="teal" vs="vs">
  <Fragment slot="a"><ul><li>Koneksi privat langsung antara dua VPC.</li><li>Non-transitif: B tidak bisa lewat A untuk menjangkau C.</li><li>Banyak VPC berarti full mesh yang sulit dikelola.</li><li>Cocok untuk jumlah VPC sedikit dan hubungan sederhana.</li></ul></Fragment>
  <Fragment slot="b"><ul><li>Hub pusat tempat banyak VPC dan on-premises attach.</li><li>Transitive routing: semua spoke bisa saling menjangkau lewat hub.</li><li>Tiap VPC cukup satu attachment, jauh lebih sederhana di skala besar.</li><li>Cocok untuk banyak VPC, banyak account, dan koneksi hybrid.</li></ul></Fragment>
</Compare>

<Box variant="analogy" icon="🚉" label="Analogi transportasi"><p>VPC Peering seperti membangun jalan langsung antara dua kota; kalau ada banyak kota, jalannya jadi banyak sekali. Transit Gateway seperti membangun satu stasiun pusat: tiap kota cukup punya satu jalur ke stasiun, lalu bisa pergi ke kota mana saja lewat sana.</p></Box>

<Box variant="exam" icon="🎯" label="Pola soal favorit"><p>Dua VPC saja yang perlu terhubung privat: VPC Peering. Banyak VPC dan on-premises yang perlu hub terpusat dengan transitive routing: Transit Gateway. Kata kunci "non-transitif" dan "full mesh" mengarah ke peering; "hub", "cloud router", dan "simplify connectivity" mengarah ke Transit Gateway.</p></Box>

</Section>

<Section num="10" id="hybrid-connectivity" title="VPN dan Direct Connect" sub="Menghubungkan AWS dengan kantor, data center, dan user remote">

<p class="lead">Hybrid connectivity berarti AWS tidak berdiri sendiri, tetapi tersambung ke jaringan perusahaan atau user remote.</p>

**[AWS Site-to-Site VPN](https://docs.aws.amazon.com/vpn/latest/s2svpn/VPNTunnels.html)** membuat koneksi aman antara jaringan on-premises dan VPC melalui tunnel IPsec terenkripsi di atas internet publik. Cocok untuk setup cepat, backup koneksi, atau kebutuhan yang tidak menuntut jalur dedicated. Tiap koneksi menyediakan dua tunnel ke dua endpoint berbeda untuk high availability, jadi jika satu tunnel bermasalah, yang lain tetap melayani. **AWS Client VPN** dipakai untuk remote user, misalnya karyawan dari laptop yang perlu masuk ke resource VPC secara aman. Keduanya berada di bawah payung **AWS VPN**.

**[AWS Direct Connect](https://aws.amazon.com/directconnect/)** menghubungkan jaringan internal ke AWS melalui koneksi privat dedicated di lokasi Direct Connect, tidak melewati internet publik. Karena jalurnya dedicated, performanya lebih konsisten dan latency lebih predictable. Untuk ujian, ingat kata kunci: koneksi dedicated, performa stabil, dan workload hybrid yang serius.

<Box variant="warn" icon="🔓" label="Direct Connect tidak otomatis terenkripsi"><p>Jebakan yang sering muncul: Direct Connect <strong>tidak</strong> dienkripsi secara default, berbeda dari VPN yang selalu IPsec terenkripsi. Jalurnya memang privat dan tidak lewat internet, tetapi jika kamu butuh enkripsi, kamu menambahkannya sendiri, misalnya lewat MACsec (di koneksi berkecepatan tinggi tertentu) atau dengan menjalankan VPN di atas Direct Connect. Jangan asumsikan "privat berarti terenkripsi".</p></Box>

<div class="tbl-wrap"><table>
  <thead><tr><th>Layanan</th><th>Kapan dipakai</th><th>Kata kunci ujian</th></tr></thead>
  <tbody>
    <tr><td>Site-to-Site VPN</td><td>Kantor atau data center ke AWS lewat internet terenkripsi</td><td>Cepat, IPsec, encrypted tunnel, dua tunnel HA</td></tr>
    <tr><td>Client VPN</td><td>User remote ke VPC</td><td>Karyawan remote, laptop, akses privat</td></tr>
    <tr><td>Direct Connect</td><td>Koneksi dedicated dari jaringan perusahaan ke AWS</td><td>Dedicated, predictable, private, tidak lewat internet</td></tr>
  </tbody>
</table></div>

<Box variant="analogy" icon="🚇" label="Analogi transportasi"><p>Site-to-Site VPN seperti jalur aman terenkripsi di jalan umum yang ramai, Client VPN seperti kartu akses pribadi untuk masuk gedung dari mana saja, Direct Connect seperti rel khusus milik perusahaan langsung ke AWS.</p></Box>

<Box variant="exam" icon="🎯" label="VPN vs Direct Connect"><p>Jika soal menyebut koneksi cepat disiapkan dan terenkripsi lewat internet, pilih VPN. Jika menyebut dedicated private connection, performa konsisten, dan tidak lewat internet, pilih Direct Connect. Ingat: VPN terenkripsi otomatis, Direct Connect tidak.</p></Box>

</Section>

<Section num="11" id="global-application" title="Global Accelerator dan API Gateway" sub="Routing global dan front door API">

<p class="lead">Tidak semua percepatan global berarti caching, kadang yang dibutuhkan adalah routing jaringan yang lebih optimal atau API front door.</p>

**[AWS Global Accelerator](https://docs.aws.amazon.com/global-accelerator/latest/dg/introduction-how-it-works.html)** membantu meningkatkan availability dan performance aplikasi untuk user lokal maupun global. Ia mengalokasikan dua static anycast IPv4 (dari dua network zone independen) sebagai pintu masuk global, lalu mengarahkan traffic lewat AWS global network ke endpoint yang sehat dan optimal. Global Accelerator beroperasi di Layer 4 (TCP dan UDP) dan dilindungi AWS Shield Standard secara default. Cocok untuk aplikasi gaming (UDP), IoT (MQTT), VoIP, atau workload yang butuh IP statis global, failover regional cepat, dan traffic yang tidak bisa dicache.

Kenapa static anycast IP penting? Karena beberapa skenario tidak bisa mengandalkan DNS yang berubah-ubah, misalnya whitelisting IP di firewall pelanggan atau aplikasi yang menyimpan IP secara hardcode. Dengan dua IP tetap, kamu punya entry point stabil sementara AWS mengatur jalur tercepat di belakang layar.

<Figure><VPCNetworkingFig07 /><Fragment slot="caption"><b>Gambar 7.</b> CloudFront (Layer 7, caching konten di edge) dibandingkan Global Accelerator (Layer 4, optimasi network path dengan static anycast IP, tanpa caching).</Fragment></Figure>

<Compare aLabel="CloudFront &middot; CDN, caching" bLabel="Global Accelerator &middot; network path" aTone="red" bTone="blue" vs="vs">
  <Fragment slot="a"><ul><li>Layer 7 (HTTP dan HTTPS), berfokus pada konten.</li><li>Menyimpan salinan konten (caching) di edge location.</li><li>Cocok untuk gambar, video, file download, static website, API cacheable.</li><li>Domain CloudFront, bukan static IP tetap.</li></ul></Fragment>
  <Fragment slot="b"><ul><li>Layer 4 (TCP dan UDP), berfokus pada jalur jaringan.</li><li>Tidak menyimpan cache, hanya mengoptimalkan rute ke endpoint sehat.</li><li>Cocok untuk gaming, VoIP, IoT, dan traffic non-cacheable.</li><li>Memberi dua static anycast IP sebagai entry point global.</li></ul></Fragment>
</Compare>

**[Amazon API Gateway](https://docs.aws.amazon.com/apigateway/latest/developerguide/welcome.html)** adalah layanan managed untuk membuat, memublikasikan, memelihara, memonitor, dan mengamankan API. Ia mendukung REST API, HTTP API, dan WebSocket API, lengkap dengan throttling, otentikasi, dan stages. Dalam peta networking CLF-C02, API Gateway berperan sebagai front door untuk API, sering terhubung ke AWS Lambda, service berbasis container, EC2, atau backend HTTP. Ia bukan load balancer umum dan bukan accelerator.

  </tbody>
</table></div>

<Box variant="exam" icon="🧠" label="CloudFront vs Global Accelerator"><p>CloudFront unggul saat konten bisa dicache atau dioptimalkan di edge. Global Accelerator unggul saat aplikasi butuh static anycast IP, routing global, dan failover endpoint untuk traffic non-cacheable.</p></Box>

</Section>

<Section num="12" id="skenario-hands-on" title="Skenario dan Walkthrough Konseptual" sub="Latihan berpikir seperti arsitek cloud pemula">

<p class="lead">Bagian ini bukan lab klik per klik, tetapi simulasi desain yang membantu kamu membaca soal skenario.</p>

<h3>Skenario 1, aplikasi web sederhana</h3>

<Steps>
  <Step><b>Buat VPC di satu Region</b><p>Pilih rentang IP privat, lalu siapkan subnet di minimal dua Availability Zone untuk pola high availability.</p></Step>
  <Step><b>Tempatkan ALB di public subnet</b><p>ALB menerima traffic HTTPS dari internet, sedangkan application server tetap berada di private subnet.</p></Step>
  <Step><b>Tempatkan database di private subnet</b><p>Database tidak perlu public IP dan hanya menerima traffic dari security group aplikasi.</p></Step>
  <Step><b>Gunakan Route 53 untuk domain</b><p>Record DNS mengarah ke CloudFront atau ALB sesuai desain akses publik.</p></Step>
  <Step><b>Tambahkan CloudFront bila user global</b><p>CloudFront mengurangi latency dan beban origin, terutama untuk static assets dan konten yang cacheable.</p></Step>
</Steps>

<h3>Skenario 2, EC2 private subnet perlu akses S3</h3>

<Steps>
  <Step><b>Identifikasi kebutuhan traffic</b><p>EC2 private subnet perlu membaca file dari S3, tetapi tidak perlu menerima koneksi internet masuk.</p></Step>
  <Step><b>Pilih gateway VPC endpoint</b><p>Untuk S3 dari VPC, gateway endpoint membantu akses privat tanpa internet gateway atau NAT device.</p></Step>
  <Step><b>Periksa route table</b><p>Route ke prefix list S3 diarahkan ke gateway endpoint pada route table subnet terkait.</p></Step>
  <Step><b>Batasi akses dengan policy</b><p>Gunakan IAM, bucket policy, dan endpoint policy untuk mempersempit akses sesuai kebutuhan.</p></Step>
</Steps>

<h3>Skenario 3, perusahaan punya data center</h3>

<Steps>
  <Step><b>Butuh cepat dan fleksibel</b><p>Pilih Site-to-Site VPN jika kebutuhan awal adalah koneksi aman dari data center ke VPC lewat internet, terenkripsi dan cepat disiapkan.</p></Step>
  <Step><b>Butuh dedicated dan predictable</b><p>Pilih Direct Connect jika workload membutuhkan koneksi privat dedicated dengan performa lebih konsisten dan tidak lewat internet.</p></Step>
  <Step><b>Banyak VPC dan banyak lokasi</b><p>Pertimbangkan Transit Gateway sebagai hub agar topologi tidak berubah menjadi banyak koneksi satu per satu seperti VPC Peering full mesh.</p></Step>
</Steps>

<h3>Skenario 4, dua VPC vs banyak VPC</h3>

<Steps>
  <Step><b>Hanya dua VPC perlu bicara</b><p>Buat VPC Peering satu lawan satu, lalu tambahkan rute ke CIDR VPC lawan di route table masing-masing.</p></Step>
  <Step><b>Bertambah jadi banyak VPC</b><p>Begitu jumlah VPC bertambah dan semuanya perlu saling terhubung, full mesh peering jadi sulit. Beralih ke Transit Gateway sebagai hub.</p></Step>
  <Step><b>Ingat sifat transitif</b><p>Peering non-transitif, jadi B tidak otomatis menjangkau C lewat A. Transit Gateway transitive, jadi semua spoke saling terhubung lewat hub.</p></Step>
</Steps>

<h3>Latihan observasi aman di Console</h3>

<Steps>
  <Step><b>Cek apakah subnet public</b><p>Buka VPC &rarr; Subnets, pilih subnet, lihat tab Route table. Ada rute 0.0.0.0/0 ke internet gateway berarti public; tidak ada berarti private.</p></Step>
  <Step><b>Lihat sifat allow-only security group</b><p>Buka EC2 &rarr; Security Groups, perhatikan hanya ada Inbound dan Outbound rules tanpa kolom deny, menegaskan SG hanya bisa allow.</p></Step>
  <Step><b>Bandingkan dengan NACL</b><p>Buka VPC &rarr; Network ACLs, lihat kolom Rule number dan Allow/Deny, menegaskan NACL stateless dengan urutan rule dan deny eksplisit.</p></Step>
</Steps>

<Recap title="Pola desain yang sehat"><ul><li>Komponen publik seminimal mungkin, biasanya load balancer atau CloudFront.</li><li>Workload dan database lebih aman di private subnet.</li><li>Gunakan security group sebagai kontrol utama dan NACL sebagai guardrail tambahan.</li><li>Gunakan endpoint untuk akses privat ke layanan AWS jika sesuai, gateway endpoint untuk S3 dan DynamoDB.</li><li>Gunakan Route 53 untuk DNS, CloudFront untuk CDN, Global Accelerator untuk routing global non-cacheable.</li><li>Dua VPC pakai peering, banyak VPC pakai Transit Gateway.</li></ul></Recap>

</Section>

<Section num="13" id="kesalahan-jebakan" title="Kesalahan Umum dan Jebakan Soal" sub="Hal yang sering membuat pemula salah pilih layanan">

<p class="lead">Banyak jawaban salah muncul bukan karena tidak tahu layanan, tetapi karena mencampur fungsi layanan.</p>

<CardGrid cols={2}>
  <Card><h4>DNS dianggap CDN</h4><p>Route 53 mengarahkan nama domain, tetapi tidak menyimpan cache konten seperti CloudFront.</p></Card>
  <Card><h4>CDN dianggap private network</h4><p>CloudFront mempercepat delivery dari edge, tetapi bukan pengganti VPC, subnet, atau security group.</p></Card>
  <Card><h4>Private subnet dianggap tidak bisa keluar sama sekali</h4><p>Private subnet bisa melakukan outbound melalui NAT gateway atau VPC endpoint, tergantung tujuan traffic.</p></Card>
  <Card><h4>Security group dan NACL tertukar</h4><p>Security group stateful di level resource, NACL stateless di level subnet.</p></Card>
  <Card><h4>PrivateLink dianggap VPN</h4><p>PrivateLink menghubungkan VPC ke LAYANAN secara privat, VPN dan Direct Connect menghubungkan JARINGAN seperti kantor atau data center ke VPC.</p></Card>
  <Card><h4>Direct Connect dianggap selalu terenkripsi otomatis seperti VPN</h4><p>Direct Connect tidak dienkripsi secara default. VPN selalu IPsec terenkripsi, Direct Connect privat tetapi butuh MACsec atau VPN overlay jika ingin enkripsi.</p></Card>
  <Card><h4>Internet Gateway dan NAT Gateway tertukar</h4><p>Internet Gateway dua arah untuk public subnet. NAT Gateway hanya outbound dari private subnet, tidak menerima koneksi masuk dari internet.</p></Card>
  <Card><h4>VPC Peering dianggap transitif</h4><p>Peering non-transitif, B tidak bisa lewat A menjangkau C. Untuk banyak VPC dan transitive routing, pakai Transit Gateway sebagai hub.</p></Card>
  <Card><h4>Gateway dan interface endpoint disamakan</h4><p>Gateway endpoint gratis, hanya S3 dan DynamoDB, lewat route table, bukan PrivateLink. Interface endpoint berbayar, banyak layanan, lewat ENI, memakai PrivateLink.</p></Card>
  <Card><h4>OAI dianggap satu-satunya cara</h4><p>OAI adalah metode legacy untuk membatasi akses S3 dari CloudFront. Pendekatan terkini yang direkomendasikan adalah OAC.</p></Card>
</CardGrid>

<Section num="14" id="ringkasan" title="Ringkasan & Tips Ujian" sub="Peta cepat untuk mengingat sebelum latihan soal">

<p class="lead">Networking di CLF-C02 adalah tentang memilih jalur, pintu, satpam, buku alamat, cabang cache, dan koneksi hybrid yang tepat.</p>

<Recap title="Poin Penting"><ul><li><strong>Amazon VPC</strong> adalah jaringan virtual terisolasi untuk resource AWS.</li><li><strong>Subnet</strong> membagi VPC, satu subnet hidup di satu Availability Zone; public vs private ditentukan oleh route table, bukan namanya.</li><li><strong>Route table</strong> mengarahkan traffic ke target seperti internet gateway, NAT gateway, transit gateway, atau VPC endpoint.</li><li><strong>Internet Gateway</strong> dua arah untuk public subnet, <strong>NAT Gateway</strong> hanya outbound dari private subnet (berbiaya per-jam plus per-GB).</li><li><strong>Security group</strong> adalah firewall stateful di level instance, allow-only, semua rule dievaluasi.</li><li><strong>NACL</strong> adalah firewall stateless di level subnet, allow plus deny, rule dievaluasi berurutan by number.</li><li><strong>Gateway endpoint</strong> gratis untuk S3 dan DynamoDB lewat route table; <strong>interface endpoint</strong> berbayar lewat ENI memakai PrivateLink.</li><li><strong>AWS PrivateLink</strong> menghubungkan VPC ke LAYANAN secara privat tanpa public IP, internet gateway, atau NAT.</li><li><strong>Route 53</strong> adalah DNS, domain registration, routing policy (delapan jenis), dan health checking.</li><li><strong>CloudFront</strong> adalah CDN (Layer 7, caching di edge); pakai OAC untuk membatasi akses S3, free tier 1 TB out per bulan always free.</li><li><strong>Site-to-Site VPN</strong> tunnel IPsec terenkripsi lewat internet (dua tunnel HA); <strong>Direct Connect</strong> dedicated private connection, tidak terenkripsi default.</li><li><strong>VPC Peering</strong> satu lawan satu non-transitif; <strong>Transit Gateway</strong> hub-and-spoke transitive untuk banyak VPC dan on-premises.</li><li><strong>Global Accelerator</strong> memberi dua static anycast IP dan optimasi network path Layer 4 (TCP/UDP) tanpa caching.</li><li><strong>API Gateway</strong> adalah front door managed untuk REST, HTTP, dan WebSocket API.</li></ul></Recap>

<h3>Tips menjawab soal</h3>

<ul>
  <li>Jika soal bicara nama domain, pikirkan Route 53.</li>
  <li>Jika soal bicara cache konten dekat user, pikirkan CloudFront.</li>
  <li>Jika soal bicara private isolated network, pikirkan VPC.</li>
  <li>Jika soal bicara firewall instance yang stateful, pikirkan security group.</li>
  <li>Jika soal bicara subnet firewall yang stateless dan deny rule, pikirkan NACL.</li>
  <li>Jika soal bicara akses privat ke S3 atau DynamoDB dari VPC, pikirkan gateway endpoint.</li>
  <li>Jika soal bicara layanan privat tanpa public IP, pikirkan PrivateLink atau interface endpoint.</li>
  <li>Jika soal bicara koneksi kantor ke AWS lewat internet terenkripsi, pikirkan Site-to-Site VPN.</li>
  <li>Jika soal bicara koneksi dedicated, predictable, dan tidak lewat internet, pikirkan Direct Connect.</li>
  <li>Jika soal bicara dua VPC saling terhubung satu lawan satu, pikirkan VPC Peering.</li>
  <li>Jika soal bicara banyak VPC yang perlu hub pusat dan transitive routing, pikirkan Transit Gateway.</li>
  <li>Jika soal bicara static anycast IP dan routing global aplikasi TCP atau UDP, pikirkan Global Accelerator.</li>
  <li>Jika soal bicara front door untuk API, pikirkan API Gateway.</li>
</ul>
