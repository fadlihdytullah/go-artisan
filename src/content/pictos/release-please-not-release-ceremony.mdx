A common release flow looks responsible: bump the version in `package.json` by hand, write the changelog in Confluence, merge your working branch into `develop`, cut a release branch, merge `develop` into it, then open one more PR into `master`. It ships every time. It also quietly taxes you every time.

```text
1. Bump package.json version by hand
2. Write the changelog in Confluence
3. Merge working branch → develop (PR)
4. Create a release branch
5. Merge develop → release branch
6. Merge release branch → master (PR)
7. Production deploys
```

Each step is a place to forget the version bump, to let the changelog drift away from the code that actually changed, or to keep a frozen QA branch alive long after the team stopped needing one. Without Git tags there is no single, honest marker of what production is running.

<div class="picto-callout">
  <span class="picto-callout-label">THE REAL PROBLEM</span>
  The old flow is not wrong. It just wraps manual ceremony around something CI can do reliably and never forget.
</div>

## What you are actually automating

Strip the ceremony down to four moving parts and the picture gets simple.

```text
Git tag             = the official release version
CHANGELOG.md        = release notes, living inside the repo
Conventional Commit = the input that generates those notes
Release Please      = the bot that opens the release PR, writes
                      the changelog, tags, and cuts the GitHub Release
```

Production deployment hangs off the release, not off every push to `master`. And for a private app that never gets published to npm, the `package.json` version is optional. The Git tag is the release identity, and it reads exactly the way a human expects.

```text
v1.0.0
v1.1.0
v1.1.1
```

## The shape of the new flow

The same beats stay, but the boring half becomes automatic.

```text
feature branch
  ↓ squash merge
develop
  ↓ staging deploy
develop → master  (merge commit, not squash)
  ↓
Release Please opens a release PR
  ↓ merge the PR
Git tag + GitHub Release
  ↓
production deploy
```

Feature branches are squash-merged into `develop`. The merge from `develop` into `master` keeps its individual commits. Release Please reads the meaningful `feat:` and `fix:` commits that landed on `master`, and production deploys only after a release is cut.

## Merge strategy is the load-bearing choice

This is the one decision the whole workflow rests on, and it points in two different directions depending on where you are.

**Feature into develop: squash and merge.** You want one clean commit per ticket, not the noisy `wip`, `fix typo`, `oops` trail from the branch. That single squash title is what the changelog will quote, so it must be a Conventional Commit.

```text
# branch commits (noise)
feat(navbar): flatten Shop All mega-menu
feat(shop-all): revamp filters, hero banner & page surfaces
feat(home): reorder sections & restore Beauty Impact

# squash title that lands on develop (signal)
feat(shop-all): revamp homepage, Shop All, and navbar
```

**Develop into master: a real merge commit.** Squashing here would collapse every `feat` and `fix` into one vague `chore: merge develop to master`, and Release Please would see nothing worth releasing. A merge commit preserves all the clean commits so the bot can still read them.

<div class="picto-callout">
  <span class="picto-callout-label">THE ONE RULE</span>
  Squash small branches into develop. Never squash develop into master.
</div>

## Conventional Commits are the input

Release notes are only as good as the commit titles feeding them. A small, boring vocabulary is enough.

```text
feat(scope):     a user-facing feature or improvement
fix(scope):      a user-facing bug fix
perf(scope):     a performance improvement
refactor(scope): internal restructuring, no behavior change
chore(scope):    tooling or maintenance
docs(scope):     documentation only
style(scope):    formatting or styling cleanup
```

The prefix decides the version bump, which is the whole point of skipping manual bumps.

```text
feat:                      → minor release (1.1.0)
fix:                       → patch release (1.1.1)
feat! / BREAKING CHANGE:   → major release (2.0.0)
chore / docs / style /
refactor:                  → usually no release note
```

Write titles a stranger could read in a changelog. The good ones describe a product change; the bad ones describe nothing.

```text
# good
feat(shop-all): revamp homepage, Shop All, and navbar
fix(cart): prevent duplicate checkout submission
perf(product-list): reduce initial render cost

# bad
Neo 2260
update stuff
fix bug
merge develop
```

<div class="picto-callout">
  <span class="picto-callout-label">THE GOTCHA</span>
  If the squash title is not a Conventional Commit, Release Please ignores it. A green Action run with no release PR almost always means exactly this.
</div>

## Wiring the GitHub Action

One workflow watches `master` and does the bookkeeping. `release-type: simple` is the strategy for a generic repo that has no language version file to maintain, so `package.json` is left untouched.

```yaml title=".github/workflows/release-please.yml"
name: Release Please

on:
  push:
    branches:
      - master

permissions:
  contents: write
  pull-requests: write
  issues: write

jobs:
  release-please:
    runs-on: ubuntu-latest
    steps:
      - name: Run Release Please
        id: release
        uses: googleapis/release-please-action@v4
        with:
          release-type: simple
          target-branch: master
```

On every push to `master`, the action scans the Conventional Commits since the last release tag, then opens or updates a single Release Please PR. That PR is where `CHANGELOG.md` gets rewritten. Merge it, and the action creates the Git tag and the GitHub Release. If your production branch is `main`, swap both `master` lines for `main` — the branch name in the config must match reality.

## No previous tag? Set a baseline

Plenty of repos have years of history and zero release tags. Do not rewrite history and do not backfill old changelogs. Tag the current production branch once, and let automation take over from there.

```bash
git checkout master
git pull origin master
git tag -a v1.0.0 -m "v1.0.0 baseline release"
git push origin v1.0.0
```

Seed `CHANGELOG.md` with a single honest entry so the file exists and the story has a starting point.

```md
# Changelog

## v1.0.0

Initial baseline release. This marks the current production state
before adopting automated release notes. Everything after this point
is tracked automatically.
```

From `v1.0.0` onward, every release is generated from the commits that come after it. A finished changelog entry stays short on purpose, because it summarizes product changes rather than echoing raw commit history.

```md
## [1.1.0](https://github.com/org/repo/compare/v1.0.0...v1.1.0) (2026-06-18)

### Features

* revamp homepage, Shop All, and navbar
```

One ticket becomes one line. If two changes are truly independent, that is a sign they wanted to be two PRs.

## Deploy on release, not on every push

Pushing to `master` should not deploy production. A release being cut should. That single shift guarantees every production deploy has a tag behind it, which makes rollback and audit trivial. The cleanest way to wire it is to keep Release Please and the deploy in the same workflow, each deploy step gated by the action's `release_created` output.

```yaml title=".github/workflows/release-and-deploy.yml"
name: Release Please and Production Deploy

on:
  push:
    branches:
      - master

permissions:
  contents: write
  pull-requests: write
  issues: write

jobs:
  release-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Run Release Please
        id: release
        uses: googleapis/release-please-action@v4
        with:
          release-type: simple
          target-branch: master

      - name: Checkout release commit
        if: ${{ steps.release.outputs.release_created }}
        uses: actions/checkout@v4

      - name: Setup Node
        if: ${{ steps.release.outputs.release_created }}
        uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: yarn

      - name: Install dependencies
        if: ${{ steps.release.outputs.release_created }}
        run: yarn install --frozen-lockfile

      - name: Build
        if: ${{ steps.release.outputs.release_created }}
        run: yarn build

      - name: Deploy production
        if: ${{ steps.release.outputs.release_created }}
        run: echo "Put production deploy command here"
```

`release_created` is only `true` after the Release Please PR is merged and a release is actually cut. Every push that does not produce a release runs the first step and stops, so production never deploys ahead of its own release metadata.

## Gotchas worth pinning

A few sharp edges show up the first week and then never again once you know them.

A successful Action run does not always mean a release PR exists. If the only commits since the last tag are merge commits or non-conventional titles like `Neo 2260`, there is nothing releasable and the bot correctly opens nothing. After a squash merge, only the final squash title matters — the original branch commits are gone.

The same-workflow pattern above is not a style preference, it is a workaround. A GitHub Release created by the default `GITHUB_TOKEN` will not trigger a separate workflow that listens on `release: published`. Gating the deploy on `release_created` inside one workflow sidesteps that entirely.

<div class="picto-callout">
  <span class="picto-callout-label">TEAM RULES</span>
  Conventional Commit titles on every PR. Squash into develop, merge-commit into master. Never hand-edit the version of a private app. Tag a baseline once. Deploy on release, not on push. Split unrelated changes into separate PRs.
</div>

## The takeaway

Manual steps are not the enemy; repetitive ceremony is. Let engineers spend their judgment on the one thing only they can decide, and let the pipeline handle versioning, the changelog, the tag, and the deploy trigger.
